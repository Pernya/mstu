# Словарь данных

## Источник

Используется официальный набор Montgomery County Warehouse and Retail Sales. В нем есть месячные продажи и перемещения товаров по поставщикам, кодам товаров и типам товаров. Для локального запуска я не забираю весь датасет целиком: количество строк на месяц ограничивается переменной `RETAIL_ROWS_PER_PERIOD`, чтобы пайплайн нормально выполнялся на ноутбуке.

## Основные поля источника

| Поле | Назначение |
| --- | --- |
| `calendar_year` | календарный год |
| `cal_month_num` | номер месяца |
| `supplier` | поставщик |
| `item_code` | код товара |
| `item_description` | описание товара |
| `item_type` | тип товара |
| `rtl_sales` | розничные продажи |
| `rtl_transfers` | перемещения в retail-точки |
| `whs_sales` | складские продажи |

## Как распарсены данные

Socrata API возвращает поля как JSON-значения, поэтому в transform-слое я явно привожу год и месяц к целым числам, продажи и перемещения к `float`, текстовые поля к очищенным строкам. Пустые значения поставщика, товара, описания и типа товара заменяются резервными метками `UNKNOWN_*`. После этого строится `period_date` как первый день месяца.

Дубликаты удаляются по набору `calendar_year`, `cal_month_num`, `supplier`, `item_code`, `item_type`, `rtl_sales`, `rtl_transfers`, `whs_sales`. Это не удаляет разные товары или разные факты продаж, но убирает повтор одной и той же строки источника.

## Бизнес-ключи

| Сущность | Состав бизнес-ключа | Причина |
| --- | --- | --- |
| Период | `PERIOD`, год, месяц | период в источнике задается годом и номером месяца |
| Поставщик | `SUPPLIER`, название поставщика | отдельного supplier id в выборке нет, поэтому используется нормализованное название |
| Товар | `PRODUCT`, `item_code`, `item_description` | код товара дополняется описанием, чтобы снизить риск коллизии неоднозначных кодов |
| Факт продаж | `SALE`, год, месяц, поставщик, код товара, тип товара | связь фиксирует факт продаж товара поставщика в конкретном месяце |
| Изменение метрик | `rtl_sales`, `rtl_transfers`, `whs_sales` | hashdiff нужен для контроля изменения измеримых показателей |

Перед хешированием значения приводятся к строке, очищаются от пробелов по краям и переводятся в верхний регистр. Хеш считается через SHA-256 и обрезается до 32 символов: этого достаточно для компактных ключей в учебном DWH и сохраняет воспроизводимость между запусками.

## Схемы DWH

| Схема | Назначение |
| --- | --- |
| `raw` | нормализованные строки источника и метаданные колонок |
| `dv` | Data Vault слой с hubs, links и satellites |
| `mart` | витрины для анализа, прогнозирования и Superset |
| `ml` | метрики модели, прогноз и результаты кластеризации |

## Data Vault

| Таблица | Тип | Назначение |
| --- | --- | --- |
| `dv.hub_period` | Hub | справочник периодов |
| `dv.hub_supplier` | Hub | справочник поставщиков |
| `dv.hub_product` | Hub | справочник товаров |
| `dv.link_sales` | Link | связь периода, поставщика и товара |
| `dv.sat_product` | Satellite | описание и тип товара |
| `dv.sat_sales_metrics` | Satellite | retail sales, retail transfers и warehouse sales |

## Витрины

| Таблица | Назначение |
| --- | --- |
| `mart.dm_monthly_sales` | продажи по месяцам |
| `mart.dm_item_type_sales` | продажи по месяцам и типам товаров |
| `mart.dm_supplier_sales` | продажи по поставщикам |
| `mart.dm_ml_sales` | агрегированная основа для ML и дашбордов |
| `ml.model_metrics` | качество последней модели |
| `ml.sales_forecast` | прогноз продаж на следующий месяц |
| `ml.supplier_cluster_profiles` | портреты кластеров поставщиков |
| `ml.supplier_cluster_members` | принадлежность поставщиков к сегментам |

## Признаки прогноза

| Признак | Назначение |
| --- | --- |
| `calendar_year` | год периода |
| `cal_month_num` | номер месяца |
| `quarter` | квартал |

| `item_type` | тип товара |

| `source_rows` | количество исходных строк в агрегате |

| `sales_rolling_std_3` | стандартное отклонение продаж за три прошлых периода |
| `sales_pct_change_1` | динамика к прошлому периоду |
| `warehouse_sales_lag_1` | складские продажи прошлого периода |
| `retail_transfers_lag_1` | retail transfers прошлого периода |

## Признаки кластеризации

| Признак | Назначение |
| --- | --- |
| `log_retail_sales` | масштаб розничных продаж без доминирования крупных значений |
| `log_warehouse_sales` | масштаб складских продаж |
| `log_retail_transfers` | масштаб перемещений в retail-точки |
| `products_count` | ширина ассортимента |
| `active_months` | устойчивость присутствия поставщика в истории |
| `avg_retail_sales_per_month` | средний месячный оборот |
| `warehouse_share` | роль складского канала |
| `transfer_share` | роль retail transfers |
| `negative_sales_share` | доля корректировок и возвратов |

| `share_*` | структура продаж по крупнейшим типам товаров |

## Исключаемые признаки

Из обучения автоматически исключаются технические хеши и hashdiff, служебные поля загрузки, дата периода, целевая переменная, исходные продажи текущего периода, сырые текстовые описания, признаки с высокой долей пропусков, нулевой дисперсией и слишком высокой категориальной кардинальностью.
